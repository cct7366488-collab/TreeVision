## Context

[SPEC.md 第拾貳章](../SPEC.md) 列出 7 個尚待決議的開放問題。在進入 **Phase 1（資料收集 + 標註）** 之前，至少需先拍板前 3 項，否則：

- **未定樹種辨識範圍** → 影響 metadata 的 `species_*` 欄位是否強制、影響資料蒐集成本
- **葉片標註單位未定** → 標註規範定不下來，標註團隊無法上工
- **健康分數權重未定** → 模型驗收標準（Spearman ρ）無法基準化

本 ADR 對 7 項議題提出有利弊分析的「初版建議」，供主持人快速決策。

---

## 議題 1：v1 是否做樹種辨識？

### Options

| 選項 | 說明 | 額外成本 |
|------|------|----------|
| **A. v1 不做** | 由使用者上傳時手動填 `species_zh` | 0 |
| **B. v1 做白名單 5 種** | 訓練輕量分類模型，限定 5 個常見種 | 標註多 30%、+1 個模型訓練 |
| **C. v1 不做、v2 再加** | 同 A，但在 schema 預留欄位 | 0 |

### 建議：**選 C（v1 不做、保留欄位）**

理由：
1. 第一個專案場域樹木應已預先編號（spec 假設），人工填樹種比訓練模型可靠。
2. 樹種辨識是獨立的 CV 任務，需另一套標註與訓練資料；v1 範圍應聚焦在「樹冠 / 葉片量測」這一條主線跑通。
3. 學術文獻顯示 5 類以上的細粒度樹種辨識需要 ≥ 5000 張影像／類，遠超 Phase 1 能蒐集的數量。
4. 到 v2 再做，可以利用 v1 已蒐集的影像當作訓練集起點。

### Consequences
- `image_metadata.csv` 的 `species_zh` 仍允許 NULL，但前端表單會在使用者選擇 `tree_id` 後**自動帶入** `tree_registry.csv` 中已登記的樹種。
- v2 規劃時新增 `SpeciesId` 模型欄位於 `models/` 目錄。

## 議題 2：葉片標註單位（全葉 vs 全葉 + 葉柄）

### Options

| 選項 | 說明 | 影響 |
|------|------|------|
| **A. 只標全葉（含葉柄一體）** | 葉柄與葉身一起標 | 簡單、標註一致性高 |
| **B. 葉身 + 葉柄分開** | 各自一個 instance | 可獨立計算葉柄長／葉身面積，更學術 |

### 建議：**選 A（只標全葉），但 polygon 邊界遵守標註規範**

理由：
1. 葉柄面積佔比小，對 `leaf_area_cm²` 與健康指標的影響在誤差容忍範圍內。
2. 葉柄與葉身的視覺分界主觀（漸變寬度），跨標註者一致性差，會拖慢標註進度。
3. 病徵主要發生在葉身，葉柄獨立資訊對健康分數無顯著貢獻。
4. 若日後研究確需葉柄資料，可在子集上補標註，不影響主資料庫。

### Consequences
- `leaf_instance` 表保留單一 `polygon_json`，不另設 `petiole_json`。
- 標註規範第參章新增註記：「葉柄併入葉片實例，邊界沿葉柄外緣」。

## 議題 3：健康分數權重（全域固定 vs 樹種特化）

### Options

| 選項 | 說明 | 風險 |
|------|------|------|
| **A. 全域固定（所有樹種同一公式）** | 簡單、可比 | 可能不公平（如針葉 vs 闊葉病徵不同） |
| **B. 樹種特化（每種一組權重）** | 更精準 | 需各樹種足量樣本與專家校正 |
| **C. 兩階段** | v1 用 A，v1.5 對特定樹種推 B 並逐步切換 | 折衷 |

### 建議：**選 C（兩階段切換）**

理由：
1. v1 沒有足夠樣本支撐樹種特化校正（每個樹種需 ≥ 100 片標註葉 × 多時段）。
2. 全域權重可建立「跨樹種健康分數」的 baseline，本身有研究價值。
3. 系統設計上把權重抽成 config（`models/LeafDefect/config.yaml` 中 `health_weights`），**從第一天就支援切換**，未來不需要修改程式。
4. 可以在 `tree_daily_summary` 增加 `health_score_global` 與 `health_score_species`（後者前期為 NULL，待校正後填入），保留可比性。

### Consequences
- 權重抽成 config，不寫死在程式碼中。
- Schema 新增 `health_score_global`（v1 必填）與 `health_score_species`（v1 可空）兩個欄位。

## 議題 4：報表格式（PDF 為主 vs Word 範本）

### Options

| 選項 | 說明 |
|------|------|
| **A. PDF 為主** | 自動產出，不可改，正式 |
| **B. Word 範本 + 自動填入** | 可由使用者後製 |
| **C. PDF + 同步輸出可編輯資料表（CSV / Excel）** | 兩者並行 |

### 建議：**選 C**

理由：
1. PDF 適合正式提交、不被改動。
2. 同時輸出 CSV/Excel 讓使用者可重新繪圖、整合至自己的 Word 文件、或寫進 FMP（森林經營計畫書）的章節中。
3. 比 Word 範本更穩定（Word 樣式跨版本問題多）。

### Consequences
- `reports/` 子目錄保持當前設計（PDF 為主）。
- API `POST /reports` 增加 `format: "pdf" | "csv" | "xlsx" | "all"` 參數，預設 `all`。

## 議題 5：GPS 遮蔽粒度

### Options

| 選項 | 顯示範例 | 適合情境 |
|------|----------|----------|
| **A. 縣市級** | 「南投縣」 | 對外公開、保護珍稀樹 |
| **B. 鄉鎮級** | 「南投縣信義鄉」 | 一般學術發表 |
| **C. 不遮蔽** | 24.5234°N, 121.3456°E | 純內部使用 |

### 建議：**雙軌制**

| 角色 | 看到 |
|------|------|
| 系統管理員 | 完整 GPS（C） |
| 已驗證研究者（含貢獻資料者） | 鄉鎮級（B） |
| 公開 / 訪客 | 縣市級（A） |

理由：
1. 遵循[森林資源資料對外公開原則](https://www.forest.gov.tw/)的精神：保護珍稀物種與場域。
2. 鄉鎮級對學術與政策研究已足夠，不影響統計分析。
3. 縣市級對外宣傳、媒體報導也能用，不洩漏具體位置。
4. 透過角色 / 權限控制，技術上由 API middleware 實作，不需多套資料庫。

### Consequences
- DB 中**永遠儲存原始 GPS**，遮蔽僅在 API serialization 時做。
- API 在 response 中加 `gps_precision: "county" | "town" | "exact"` 欄位，供前端決定顯示樣式。

## 議題 7：多語系

### Options

| 選項 | 範圍 |
|------|------|
| **A. 純中文** | 介面與報表只支援繁體中文 |
| **B. 中英雙語** | 介面與報表雙語 |
| **C. 含原住民族語** | 中英 + 阿美語 / 排灣語等 |

### 建議：**選 A（v1）+ 預留 B 的 i18n 架構**

理由：
1. v1 使用者為台灣林業工作者，繁體中文需求最強。
2. 中英雙語會增加翻譯維護成本，且科學名詞（如 chlorosis、necrosis）已有公認中譯。
3. 但前端從第一天就用 i18n 框架（如 `next-intl`）寫死英文 fallback，未來加語系不需重構。
4. 原住民族語（C）不在 v1 預算內，但若獲科技部原民司支持可獨立加。

### Consequences
- 前端使用 `next-intl`，鍵值寫繁中，預留英文 key。
- 報表標題與欄位名稱以繁中為主，附英文括號注記（例如：「健康分數 (Health Score)」）。
