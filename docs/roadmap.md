## 壹、目前進度位置（一眼看懂）

整個系統分「地基層」與「產品管線」。**目前地基大致完成，產品管線一塊都還沒長出來**——還沒有任何真實影像、訓練過的模型、或可運作的平台。完成度粗估 **15~20%**，集中在最前面的「定義與地基」。

### 一、地基層（藍圖與工具）— 大致完成

| 項目 | 狀態 |
|------|------|
| 規格與決策（SPEC v0.2、ADR-0001~0005 全 accepted、PI 諮詢定案）| ✅ |
| 資料模型（18 JSON schema 四層架構、data-schema v0.2、13 CSV 模板）| ✅ |
| 田野採集 SOP（pilot-protocol v0.2 固定攝點）| ✅ |
| 標註規範（annotation-guideline v0.2）| ✅ |
| ETL 資料管線 | ◐ 調查資料→6 實體 ✓；影像管線 / DB loader ✗ |

### 二、產品管線（真實資料流）— 全部尚未開始

```
① 影像採集 → ② AI 模型訓練 → ③ 指標計算 → ④ 試驗對照分析(ANOVA) → ⑤ 儀表板/報表
   ⬜            ⬜               ⬜            ⬜                      ⬜
支撐基建：平台 / REST API / 資料庫(PostgreSQL) / 前端　⬜
```

### 三、▶ 你在這裡

**地基完成，下一步是「拍第一張影像」。** 影像採集是整條鏈的閘門——沒有影像，模型無法訓練、指標算不出、試驗對照無從比較、報表是空的。

## 貳、兩軌並行開發

| 軌道 | 主導 | 內容 | 解鎖 |
|------|------|------|------|
| **實體軌** | 你 / 合作社 | 採購設備 → 攝點建置 → 拍照 → 標註 | **解鎖整個產品鏈** |
| **工程軌** | TreeVision（可先備）| 平台骨架 / DB loader / 標註工具選型 / 訓練環境 / 報表模板 | 等影像到位即可接上 |

> 兩軌可並行：實體軌推採購與第一場拍攝的同時，工程軌先把平台地基與 DB 載入做起來，第一批影像回來即能跑通「採集→入庫→（之後）推論」。

### 階段 1B：資料收集 + 標註（2026-07 ~ 2026-10）

| 任務 | 目標數量 | 備註 |
|------|----------|------|
| 影像採集 | **≈ 1.7 萬張/年**（~480 株 × ≥3 攝點 × 12 場次）| 按樣區分批、季配/月配 |
| 標註團隊培訓 | 2 名（主標 + 複標）| 標註工具選型待定（CVAT/LabelMe/Roboflow）|
| 標註完成 | 80% 影像 | 餘留 hold-out |
| 跨標註 IoU | ≥ 0.55（病徵）、≥ 0.75（樹冠）| 達標才釋出訓練 |

### 階段 2：模型 v0 訓練（2026-09 ~ 2026-11）

| 模型 | 達成指標 |
|------|----------|
| CanopySeg | mIoU ≥ 0.75 |
| LeafInst | mAP@50 ≥ 0.65、葉計數 MAPE ≤ 15% |
| LeafDefect | 各類 IoU ≥ 0.50 |
| PlantStructure（v1+）| 株高 MAPE ≤ 10% |
| 試驗對照驗證 | 影像 vs 人工量測 r ≥ 0.7 |

### 階段 3：平台 MVP（2026-10 ~ 2027-01）

| Sprint | 主要產出 |
|--------|----------|
| S1 | 後端 skeleton + DB（PostgreSQL，schema 已就緒）+ auth |
| S2 | 影像上傳 + metadata 表單 + 品質檢查 |
| S3 | 推論佇列 + worker + 結果儲存 |
| S4 | 樹木資料庫頁 + 影像列表 + 單影像詳情 |
| S5 | 儀表板（場域 / 樹木 / **處理組對照**）|
| S6 | PDF 報表（單影像／單樹健康履歷／**試驗對照 ANOVA**）|
| S7 | 內部 UAT、效能調校 |

### 階段 4：內部試營運（2027-01 ~ 2027-03）

1~2 場域全面導入、月度回饋 → backlog、模型 v1 升級、文件補完。

### 階段 5：公測上線（2027-04 起）

開放外部試用、repo 公開、學術發表（研討會 / SCI）。

每階段結束需檢核：

- [ ] 達成標註 IoU 門檻？
